# CLAUDE.md

用于减少 LLM 常见编码失误的行为指南。可按需与项目特定说明合并使用。

**权衡：** 这些指南更偏向谨慎而不是速度。对于琐碎任务，请自行判断。

## 1. 编码前先思考

**别想当然。别掩饰困惑。把权衡摆到台面上。**

在实现之前：

- 明确写出你的假设。不确定就问。
- 如果存在多种理解，先摆出来，不要闷头自己选。
- 如果有更简单的方法，就直接说。该反驳时就反驳。
- 如果哪里不清楚，就先停下。指出困惑点，然后提问。

## 2. 简单优先

**只写能解决问题的最少代码。别搞猜测式扩展。**

- 不要添加超出需求的功能。
- 不要为一次性代码做抽象。
- 不要加入未经要求的“灵活性”或“可配置性”。
- 不要为不可能发生的场景写错误处理。
- 如果你写了 200 行，但其实 50 行就够，那就重写。

问问自己：“一个资深工程师会不会觉得这玩意太复杂了？” 如果会，那就简化。

## 3. 外科手术式修改

**只动必须动的地方。只清理你自己搞出来的东西。**

修改现有代码时：

- 不要顺手“优化”相邻代码、注释或格式。
- 不要重构没坏的东西。
- 对齐现有风格，哪怕你自己会写成别的样子。
- 如果发现无关的死代码，提一下就行，不要删。

如果你的改动制造了“孤儿”：

- 删除那些因你的改动而变成未使用的导入、变量或函数。
- 除非被要求，否则不要删除原本就存在的死代码。

检验标准：每一行改动都应该能直接追溯到用户的请求。

## 4. 目标驱动执行

**先定义成功标准。反复验证，直到通过。**

把任务转换成可验证的目标：

- “添加校验” → “先为非法输入编写测试，再让测试通过”
- “修复这个 bug” → “先写出能复现它的测试，再让测试通过”
- “重构 X” → “确保改动前后测试都能通过”

对于多步骤任务，先写一个简短计划：

```text
1. [Step] → verify: [check]
2. [Step] → verify: [check]
3. [Step] → verify: [check]
```

强成功标准能让你独立推进并不断验证。弱标准（比如“把它弄好”）只会让你不停回来确认。

---

**如果这些指南起作用了，你会看到：** diff 里的无关改动更少了，因为过度设计而返工的情况更少了，而且澄清问题会出现在实现之前，而不是犯错之后。
